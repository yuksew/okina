## P1: 検証基盤とバックテスト

**やること**
1. リポジトリ整備: pnpmモノレポ（`packages/strategy` / `apps/backtest` / `apps/worker`）
2. Tiingo APIキー取得、ヒストリカル取込スクリプト（米国ETF候補 8〜12本、15年以上＋Frankfurter為替）
3. データ品質チェック（欠損・異常値・Stooqとのクロスチェック）
4. 戦略S1〜S4（[01-strategy-spec](01-strategy-spec.md) §2）を `packages/strategy` に純粋関数で実装
5. バックテスト実行（コスト・税・為替込み）＋パラメータ感応度分析 → 結果を `docs/05-backtest-report.md` に記録
6. 採用戦略とコア:サテライト比率の決定（ユーザー承認）

**完了条件（ゲート）**: 01-strategy-spec §5.2 の合格ライン（DD≤20%・シャープ≥ベンチマーク・感応度で結論不変・週1発注以内）を満たす戦略が1つ以上あり、ユーザーが採用を承認した。

**→ 2026-07-12 ゲート通過。** 採用構成「コアS5（NISA静的配分75%）+ サテライトブレンドS2/S4（moomoo特定25%）」。詳細と誠実性の記録は [10-p1-gate-decision.md](10-p1-gate-decision.md)。

**リスク**: 全戦略が不合格の可能性は普通にある。その場合は S1（Buy&Hold+リバランス）を採用して「規律の仕組み化」に価値を置くか、条件を見直すかをユーザーが判断する。**不合格なのに「惜しいから」と基準を緩めるのが最大の失敗パターン**（ここに注意書きを残す）。

## P2: MVP運用基盤＋ペーパートレード

**やること**
0. 口座準備（ユーザー作業）: NISA口座（SBI/楽天いずれか。両社ともNISAの米国ETF売買無料）、**ウィブル証券**・特定口座（源泉徴収あり）の開設（moomooは業務停止命令で受付停止中のため変更。10-p1-gate-decision 改訂履歴）
   - **2026-07-20: SBI証券 開設済み、NISA口座は仮開設**（税務署審査中。仮開設でも取引可能だが、審査NG時は課税口座扱いに遡及するため、コアS5の買付は本開設確定後が安全）。残り: NISA本開設の確認、ウィブル証券（サテライト側）
1. GitHubプライベートリポジトリへpush（Routinesの前提条件）
2. Workers Paid契約（$5/月）、D1スキーマ構築、ヒストリカル投入
3. 日次Cronジョブ（取込→シグナル→保存）、Discord即時アラート、データ欠損アラート
4. **状況更新レイヤー**（本システムのキモ。[05-status-update-design](05-status-update-design.md)）:
   Worker API（snapshot等6エンドポイント）→ `scripts/status/` CLI → `/status-update`・`/weekly-review` スキル → Routines登録 → 番犬Cron → ゴールデンテスト4ケース
5. ダッシュボード（Hono SSR + Cloudflare Access）: ポジション・シグナル・DD・資産推移・レポート閲覧
6. 仮想ポジション記録（ペーパートレード台帳）と乖離検知
7. ペーパートレード運用開始（シグナル→状況更新→仮想発注を記録）

**完了条件（ゲート）**: 01-strategy-spec §5.3 —
バックテストとの計算一致・オペレーション無欠落（3〜6ヶ月）・スリッページがコストモデル内。

**リスク**: 通知を見ない/仮想発注をサボる（→規律が実資金でも崩れる予兆。ゲート不合格として扱う）。Tiingo無料枠の仕様変更（→Stooqフォールバックと欠損アラートで検知）。

## P3: 実資金・半自動運用

**やること**
1. 口座準備: コア=NISA成長投資枠（国内証券・手動）、サテライト=特定口座（源泉徴収あり・手動）
2. **まず30万円以下**で3ヶ月運用（シグナル→通知→人間が公式アプリで発注→ダッシュボードに約定入力）
3. 乖離検知・入力忘れ検知・DD警告の実運用確認
4. 3ヶ月問題なければ段階増額（50万→100万）。増額判断は月次レビューで明示的に行う

**完了条件（ゲート）**: 6ヶ月間、ルール逸脱ゼロ（またはすべて検知・記録済み）で運用でき、実績DD・コストがバックテスト想定と整合。

**リスク**: 実資金での心理的裁量介入（→乖離検知が拾う設計。介入したら必ず記録しレビュー）。相場急変でDD-20%到達（→戦略停止判断の通知が飛ぶ。これは「システムの失敗」ではなく設計通りの動作であることを事前に理解しておく）。

## P4: 発注自動化の再評価（任意・実装しない前提で条件だけ定義）

**着手条件（すべて満たすまで着手しない）**
1. P3を6ヶ月以上、ゲート合格で運用済み
2. 手動発注の負担が実測で問題になっている（例: 発注忘れ・遅延が月1回以上）
3. moomoo/Webull特定口座のAPI仕様・規約を再調査し、安全装置（キルスイッチ・発注上限・冪等キー・異常時自動停止）の設計レビューをユーザーが承認

**内容**: サテライト（特定口座）のみ自動化。固定IP VPS上の発注ゲートウェイ＋Workersからの署名付き指示（[03-architecture](03-architecture.md) §4）。NISAコアは手動のまま。
