## 2. 背景・判断理由

### 2.1 エッジについての正直な前提
- ユーザーは株式投資の経験・知識が深くなく、独自のエッジ仮説を持たない。
- 目的は「エッジの追求」ではなく **「投資判断の仕組み化」**。
- この前提でのエッジなし裁量スイングトレードは、コスト分だけ市場平均に負ける公算が高い。よって**裁量を排したルール型戦略**に寄せる。優位性の源泉は「独自の発見」ではなく、「学術的・実務的に知られたファクター（モメンタム等）を規律よく実行し続けること」に置く。

### 2.2 「国内証券/NISA」×「完全自動」の矛盾と解消
- 国内証券の米国株・ETF向け発注APIは実質存在しない（フェーズ1でweb裏取り予定）。NISA口座への自動発注はさらに困難。
- ユーザー判断: **税務メリット優先。自動化は半自動（通知→手動発注）で妥協**。
- 資金〜100万円の規模では、NISA非課税メリットが完全自動化のメリットを上回るという判断。
- 将来資金規模が拡大し完全自動の価値が増した場合は、API対応口座（IBKR等）への移行を再検討する（ロードマップに再評価ポイントとして記載する）。

### 2.3 売買頻度への含意
- 半自動（人間が発注）である以上、**発注アクションは週次程度に収まる戦略**が現実的。日中の高頻度シグナルは運用が破綻する。
- → 戦略候補は「日次データで計算、月次〜週次でリバランス/ローテーション」の帯を主軸とする。

## 3. 「状況を最新に更新」の定義（全部入り）

定期実行で更新・提示する内容:

1. **保有ポジションの損益・リスク** — 評価額、損益、損切りライン/ルール逸脱までの距離
2. **シグナル検知** — 戦略ルールに基づくエントリー/イグジット/リバランス候補の自動検出
3. **相場環境サマリー** — 指数・セクター動向、VIX、金利などマクロ要約
4. **プッシュ通知** — 異常時・シグナル発生時にスマホ等へ通知

## 4. 制約・原則（プロジェクト共通、再掲）

- 検証優先: バックテスト → ペーパートレードで優位性（少なくとも規律的に運用可能なこと）を確認するまで実資金を入れない。
- 段階導入: 手動判断支援 → 半自動。フェーズをスキップしない。
- 利益は保証されない前提。Claudeは金融アドバイザーではなく、特定銘柄の推奨は行わない。
- 半自動でも安全側: シグナルの妥当性チェック、通知の冪等性、データ異常検知を持たせる。
