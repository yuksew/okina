## 1. 証券会社API・NISA自動売買の現状

### 結論
- **「NISA口座で米国株・米国ETFをAPI自動発注」できる公認手段は確認できなかった。**
- ただし「国内証券に個人向けAPIがない」は誤り。**特定口座＋公認APIで米国株を自動発注できる国内証券は存在する**（moomoo・ウィブル）。→ フェーズ0で確定した「半自動止まり」の判断は維持しつつ、将来の自動化オプションは残る。

### 各社の状況

| 証券会社 | 個人向け株式API | 米国株 | NISA×API | 確度 |
|---|---|---|---|---|
| SBI / マネックス / 松井 | なし（株式） | — | — | 消極的事実（裏取り不可の性質） |
| 楽天証券 | マーケットスピードII RSS（Excel/Windows専用、公認、発注可） | **対象外（国内株のみ）** | 国内株現物は**NISA成長投資枠の口座区分あり**（一次） | 一次 |
| auカブコム（三菱UFJ eスマート） | kabuステーションAPI（REST、公認） | **非対応**（公式仕様書で一次確認） | **不可**（AccountTypeにNISAなし、一次確認） | 一次 |
| **moomoo証券** | Moomoo API（公認、自動売買可、追加手数料なし） | **対応**（中心market） | 日本株はNISA対応の公式言及（準一次）。**米国株×NISAは裏取り不可** | 一次/準一次 |
| **ウィブル証券** | Webull OpenAPI（申請制、公認）。日米両対応は国内唯一 | **対応**（現物/信用/24h/端株） | **NISA口座自体を取扱いなし**（一次） | 一次 |
| IBKR (IBSJ) | TWS API等（公認、成熟） | 対応（全市場） | **特定口座なし=確定申告必須**（一次）。NISAは2025年7月開始、API発注可否は裏取り不可 | 一次 |

### 設計への含意
1. コア（NISA・低回転）: 国内証券で**手動発注**が確定路線。楽天ならRSS経由の国内株NISA発注は可能だが、米国ETFは対象外なので効果は限定的。
2. サテライト（特定口座・ローテーション）: 手動発注で開始。**将来自動化するならmoomoo/Webullの特定口座が現実的な移行先**（確定申告不要のまま自動化できる）。フェーズ0の「税務優先 vs 完全自動」のトレードオフは想定より緩い。
3. 注意: moomoo APIはOpenDゲートウェイ（常駐プロセス）経由の構成が基本 → 自動化する場合はWorkersでなくVPS等が必要になる見込み（§3のegress IP問題とも整合）。※ゲートウェイ要件の詳細は自動化フェーズ前に再調査

## 2. データソース（無料方針との整合）

### 採用（推奨）
| 用途 | ソース | 内容 | 確度 |
|---|---|---|---|
| 米国ETF/株 日次OHLCV | **Tiingo Free** | 30年超ヒストリカル、**配当分割調整済み（adjClose）**、1,000req/日・500シンボル/月・50req/時。公式REST、規約クリーン、Workersから直接叩ける | 一次 |
| USD/JPY 日次 | **Frankfurter API** | ECB参照レート、キー不要・無料・1999年〜のヒストリカル | 一次 |
| 品質クロスチェック | Stooq | 無料CSV・30年超。ただし調整手法非公開・日次取得上限低 → 主ソース不適、検証用。**追記（2026-07-10実測）: CSVエンドポイントがJSボット検証を返し、サーバーサイド取得は不可。ブラウザ手動DLのみ可** | 一次/二次＋実測 |

### 日本上場ETFの扱い
- **合法・無料で10年分の日次データを取る手段は存在しない**（Yahoo!ファイナンスJP・株探はスクレイピング明確禁止）。
- 正攻法はJPX公式の **J-Quants API**（Light ¥1,650/月で5年・日次運用可 / Standard ¥3,300/月で10年。Freeは12週遅延で運用不可）。ただし**AdjustmentCloseは配当未調整**でトータルリターン検証には分配金補完が必要。
- → **MVPは米国ETFのみに絞る**（無料方針と整合）。日本上場ETFは「J-Quants課金の価値があるか」をフェーズ2以降で再評価。

### 除外（理由つき）
- Yahoo非公式API/yfinance: 規約違反リスク＋WorkersのデータセンターIPはブロック対象になりやすい
- Alpha Vantage無料: 25req/日、adjusted dailyがプレミアム化
- Finnhub無料: ヒストリカルローソク足が有料化（403）
- Polygon（→Massiveへリブランド）: 無料枠の深度不足＋料金流動的（裏取り不可）

## 3. Cloudflare実行モデルの制約（公式docs一次、2026-07-10時点）

| 項目 | 事実 | 本システムへの含意 |
|---|---|---|
| プラン | 無料はCPU 10ms → 指標計算不能 | **Workers Paid（$5/月）前提** |
| Cron Triggers | 最小1分粒度。Paidは間隔≥1時間でCPU 15分。発火精度・遅延のSLAは**なし（裏取り不可）** | 日次バッチには十分。秒精度が要る処理には使わない |
| DO Alarms | **at-least-once＋最大6回リトライを公式明記** | 時刻依存・確実性が要る処理はCronでなくAlarmsへ |
| メモリ / 同時接続 | 128MB/isolate、同時アウトバウンド6本 | 全時系列オンメモリ不可 → チャンク処理。並列fetchは6本まで |
| **egress IP** | **固定不可（専用IPはEnterprise限定）** | IP許可制の外部API（一部証券API）はWorkers単体では詰む。自動発注フェーズはVPS/Fly.ioハイブリッド設計 |
| アウトバウンドWebSocket | hibernation不可＋15分制限 → 常時接続不向き | リアルタイムティック監視はやらない（要件的にも不要） |
| D1 | 10GB/DB。日次OHLCVは2,000銘柄×20年でも余裕。読み取り課金はフルスキャンで爆発 → (symbol, date)複合PK必須 | 主ストアとして適合 |
| Queues | at-least-once（二重配信あり得る）、遅延最大24h | MVP規模（ETF約10本）では不要。使う場合は冪等キー必須 |
| 常駐/ポーリング | 不可。公式代替: Cron/Alarms/Queues/Workflows/Containers | 日次〜週次要件なら問題にならない |

## 4. Claude Code Routines（スケジュール実行）

- **実在する**（2026年4月リリース、Research Preview）。クラウド実行（マシンオフ可）、Proプランで**5回/日・最小間隔1時間**。環境変数でシークレット、Network Access制御、webhook失敗通知（実験的）。
- 適性: **AI による要約・レポート生成**（相場環境サマリー、週次レビュー）には好適。
- 限界: Research Previewで仕様変更リスク。実行時間上限は非公開（裏取り不可）。**決定論的であるべき処理（シグナル計算・データ取込）をLLM実行に任せるのは不適** → その部分はWorkers Cronが担う。

## 5. 主な裏取り不可項目（推測で埋めていない）

- moomoo APIの米国株取引におけるNISA口座可否 / IBKRのNISA口座でのAPI発注可否
- Cron Triggersの発火精度・遅延保証、リトライ回数
- 非EnterpriseでのIP許可制API向けのCloudflare公式回避策
- Claude Code Routinesの実行時間上限
- Tiingoの対応取引所の完全一覧（東証非対応は状況証拠のみ）
- Stooqの利用規約明文（商用・大量取得の可否）
